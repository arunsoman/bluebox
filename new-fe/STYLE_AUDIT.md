# Style Audit — naming consistency across `new-fe/src`

Generated by reading every `.tsx` page/component and its matching `.module.css` (89 components, all of `src/components/**` + `src/pages/**`). Goal: find where the same visual/semantic UI role (a modal header, a status pill, an input wrapper, etc.) has a different class name in different files, so a shared vocabulary can be adopted.

## 0. Read this first: the class-naming problem is downstream of a token problem

Before any renaming, there's a more urgent issue: **the app is running on (at least) three disconnected CSS-variable vocabularies, and two of them are entirely undefined.**

`src/styles/tokens.css` defines exactly one system — a dark "neon terminal" palette: `--term-bg*`, `--term-fg*`, `--term-border*`, `--color-primary` (`#00ff9d`), `--color-secondary`, `--color-success/-warning/-error` (+ `-dim`/`-glow`), `--color-layer-*`, `--shadow-3d*`, `--focus-glow-*`, `--font-mono`/`--font-ui`, `--z-modal`, `--z-panel`.

Three other groups of components were evidently built against a *different*, light-themed token set that **does not exist anywhere in the repo** (confirmed by grepping all of `src/`):

| Group | Components | Undefined vars used | Renders how? |
|---|---|---|---|
| **"Light vocabulary A"** | All `onboarding/*` (except LandingScreen), `dashboard/NewProjectModal`'s wrapped `common/Modal`, `chat/*`, `steering/*`, `files/*`, `editor/*`, `nodes/*` + `gate/*` (via `EditorForms.module.css`) | `--color-white`, `--color-silver`, `--color-cloud`, `--color-ink`, `--color-slate`, `--color-frontend`, `--color-error-bg`, `--z-dropdown`, `--z-tooltip` | **No fallback given** → resolves to nothing / inherits. This is a live rendering bug in a large fraction of the app, not just a naming nit. |
| **"Light vocabulary B"** | `graph/BlueprintGraph*`, `audit/AuditPanel` | `--color-border`, `--color-text-secondary`, `--color-text-tertiary`, `--color-surface-active`, `--color-surface-subtle`, `--color-primary` (reused with a *different* intended value) | Every usage carries a **hex fallback** (`var(--color-primary, #2563eb)` etc.), so these render, but with a light, blue-accent palette that doesn't match the rest of the app at all. |
| **"Hardcoded surface system"** | `common/AiConfigModal`, `common/LogViewerModal` | `var(--surface-high, #1e1e2e)` and similar — kebab-case class keys (`styles['ai-config-modal-overlay']`), not camelCase like everywhere else | Renders via hardcoded hex fallback, dark-purple-accent palette, distinct from both the terminal theme and vocabularies A/B. |

Net effect: there isn't "the app's design system" plus a few outliers — there are **four** different visual languages live simultaneously (terminal/neon, light-A, light-B, dark-purple), and two of the four (A, and the share of B without fallbacks) are partially broken because their tokens were never added to `tokens.css`. Standardizing *class names* on top of this will just propagate four palettes under shared names. Recommend resolving which palette is canonical and back-filling `tokens.css` (or rewriting the consumers) before/alongside the rename pass below.

## 2. Patterns that are *already* consistent — don't touch these

- `card`, `linkButton`, `progressTrack`/`progressFill`, `checkboxRow`, `panel` (name, not the duplicated CSS), `name` (entity title label across NodeDetailCard/BookmarkDrawer/DraftNodeRow), `tab`/`active` (PannelChrome+EditorTabs agree).
- `EditorForms.module.css` is shared verbatim by **13 components** (`*EditorForm.tsx` ×5, `StringListField`, `ValidationStatusList`, `AddNodeModal`, `DeleteNodeModal`, `NodeEditorModal`, `RestoreRevertModal`, `EnrichmentPanel`, plus `gate/`'s `BulkFixWizard`, `CompletenessGateModal`, `NodeValidationInspector` importing it cross-directory) with zero divergence — the best example of reuse in the codebase, worth treating as the template for consolidating everything else in this report.
- `ActorDiscoveryPanel`, `CapabilityDefinitionPanel`, `UseCaseDecompositionPanel`, `GenericStagePanel`, `StoryDecompositionPanel`, `TaskDecompositionPanel` all consume `StagePanelChrome` purely through props — zero duplicated header/title/badge markup. Clean composition, no fix needed.

## 3. Suggested order of work

1. **Resolve the token-vocabulary split** (§0) — either back-fill `tokens.css` with the light-theme neutrals (`--color-white/-silver/-cloud/-ink/-slate/-frontend`, `--color-border/-text-secondary/-text-tertiary/-surface-*`) or migrate those consumers onto the existing dark/neon tokens. Do this before renaming classes, since a rename on top of dead variables just relabels the bug.
2. **Add the missing `modalFooter` slot to `Modal.tsx`** — collapses ~12 inline-style duplicates and 4 ad-hoc class names into one, the single highest-leverage fix here.
3. **Unify the badge/pill family** — biggest source of duplicate names (10+ variants for one role) and the one place where the *same data* (e.g. risk classification) gets visually different treatment (pill vs. plain text) depending on which component renders it.
4. Work through the rest of §1 top-to-bottom — most are pure renames with no behavior change.
